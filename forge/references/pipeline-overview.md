### Phase Flow (linear, sequential)

```
Discovery → Product → Design → Architecture → Specs → Implementation → Quality → Deploy
```

Each phase MUST be completed (all artifacts `approved`) before advancing to the next.

## 4. CLAUDE.md Template (generated by `forge:init`)

This template is written to the project root by `forge:init`. It establishes the pipeline as law for all Claude interactions within the project.

```markdown
# CLAUDE.md

## Project: <PROJECT_NAME>

This project follows the **Forge Pipeline** — 8 sequential phases that transform an idea into production software.

## Pipeline Rules (Non-Negotiable)

1. **Pipeline is law.** Every artifact must be produced in order. No skipping phases.
2. **Artifacts are mandatory.** Each phase has required deliverables. All must reach `approved` status.
3. **Context accumulates.** Each phase reads ALL previous artifacts. Nothing is forgotten.
4. **Decisions are documented.** Every significant decision gets an ADR.
5. **Formats are standard.** Diagrams in Mermaid. Data in YAML/JSON. Docs in Markdown.
6. **Quality is built in.** Security, accessibility, and testing are not afterthoughts.
7. **Communication is clear.** After each artifact, summarize what was done and what comes next.

## Current State

State is tracked in `.status` (YAML). Use Forge commands to interact:

| Command | Purpose |
|---------|---------|
| `/forge:status` | View current progress |
| `/forge:next` | Execute next pending artifact |
| `/forge:phase <name>` | Detail a specific phase |
| `/forge:review` | Review current phase artifacts |
| `/forge:advance` | Close current phase, open next |

## Phase Order

0. **Discovery** — Problem Statement + Competitive Landscape
1. **Product** — PRD + User Story Map + Shaped Pitch + MVP Definition
2. **Design** — User Flows + Wireframes + Design System + Tokens
3. **Architecture** — C4 Diagrams + ADRs + Data Model
4. **Specs** — Feature-level Technical Specifications
5. **Implementation** — Code Standards + Implementation Plan + Code
6. **Quality** — Test Strategy + QA Checklist + Security Baseline
7. **Deploy** — Deploy Strategy + Observability Setup

## Directory Structure

```
00-discovery/          # Phase 0 artifacts
  research/            # Supporting research
01-product/            # Phase 1 artifacts
02-design/             # Phase 2 artifacts
  wireframes/          # One file per screen
03-architecture/       # Phase 3 artifacts
  adrs/                # Architecture Decision Records
04-specs/              # Phase 4 artifacts
  features/            # One spec per feature
05-implementation/     # Phase 5 artifacts
  src/                 # Source code
06-quality/            # Phase 6 artifacts
07-deploy/             # Phase 7 artifacts
  infra/               # Infrastructure configs
```
```

## 5. Fundamental Rules

These rules govern the entire Forge pipeline and must be enforced by all commands and skills.

### 5.1 Pipeline is Law
- Phases are sequential: 0 → 1 → 2 → 3 → 4 → 5 → 6 → 7
- A phase cannot start until the previous phase has status `completed`
- Exception: Phase 0 (Discovery) starts immediately after `forge:init`

### 5.2 Artifacts are Mandatory
- Each phase defines required artifacts in `.status`
- Every artifact follows a lifecycle: `pending` → `draft` → `approved`
- A phase cannot be `completed` until ALL its artifacts are `approved`

### 5.3 Context Accumulates
- Every skill/agent MUST read all previous artifacts before producing new ones
- Cross-references between artifacts must be explicit and correct
- The `forge:review` command verifies inter-phase coherence

### 5.4 Decisions are Documented
- Any significant technical decision must be recorded as an ADR
- ADRs are numbered sequentially and immutable after approval
- New decisions that override previous ones create a new ADR with "Supersedes ADR-NNN"

### 5.5 Standard Formats
- **Diagrams:** Mermaid (flowchart, erDiagram, C4Context, etc.)
- **Structured data:** YAML (`.status`) or JSON (design tokens)
- **Documents:** Markdown with clear sections and tables
- **Naming:** kebab-case for files and directories

### 5.6 Quality is Built-In
- Security (OWASP Top 10) is checked, not bolted on
- Accessibility (WCAG 2.1) is considered from Design phase
- Testing strategy is defined before code is written

### 5.7 Communication is Clear
- After producing any artifact, summarize what was done
- Suggest the next action (e.g., "Run `/forge:review` to validate")
- Never leave the owner wondering what to do next

## 6. Artifact Execution Order Within Phases

When `forge:next` is invoked, it follows this order within each phase:

| Phase | Execution Order |
|-------|----------------|
| 0 — Discovery | 1. `problem-framing` → 2. `competitive-landscape` |
| 1 — Product | 1. `product-definition` → 2. `scope-shaper` |
| 2 — Design | 1. `user-flow-mapping` → 2. `wireframe-spec` → 3. `design-system-bootstrap` |
| 3 — Architecture | 1. `architecture-design` (triggers `adr-writer`) → 2. `data-model-designer` |
| 4 — Specs | 1. `tech-spec-writer` (1 per feature) |
| 5 — Implementation | 1. `code-standards` → 2. `dev-orchestrator` |
| 6 — Quality | 1. `test-strategy` → 2. `qa-checklist` → 3. `security-baseline` |
| 7 — Deploy | 1. `deploy-strategy` → 2. `observability-setup` |
